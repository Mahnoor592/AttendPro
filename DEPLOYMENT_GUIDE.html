1. How the deployed app fits together

Your app has three separate pieces. In production each lives in a different place, and they talk to each other over the internet.

Frontend
React (Vite) → Vercel
The website users see. A static build (HTML/JS/CSS) hosted on Vercel with automatic HTTPS.
Backend / API
Laravel 13 → Railway
Handles login, geofence math, attendance rules, all data. Exposes the /api endpoints.
Database
MySQL → Railway
Stores users, branches, schedules, attendance logs. A managed service Railway runs for you.

The flow: Browser loads the site from Vercel → the site calls the API on Railway → Laravel reads/writes the MySQL database (also on Railway) → the response goes back to the browser.

Why this split Frontend and backend are hosted separately because they're different kinds of app (static files vs. a PHP server). This is the standard, free-tier-friendly setup — and it means you can update one without touching the other.

2. Before you start

  • A GitHub account and your project pushed to a repo (Part A).
  • A Railway account (sign up with GitHub at railway.app).
  • A Vercel account (sign up with GitHub at vercel.com).
  • Git installed locally and working (git --version succeeds).
Give yourself time Deploy at least a day before any demo. First deployments almost always need one or two small fixes (usually a URL or a setting) — that's normal, and this guide lists every common one.

3. Part A — Push code to GitHub

Run these from the project root (the folder containing both backend and frontend). This was already done during setup; included here for reference.

cd "C:\Users\...\DBMS Project\Project"
git init
git branch -M main
git add .
git commit -m "Initial commit - AttendPro attendance system"
git remote add origin https://github.com/YOUR-USERNAME/YOUR-REPO.git
git push -u origin main
Never commit secrets Your backend/.env, node_modules, and vendor must NOT be pushed. The .gitignore files already exclude them. After git add ., run git status and confirm none of those appear.

4. Part B — Backend + MySQL on Railway

1

Create the project

Go to railway.app → New Project → Deploy from GitHub repo → choose your repo. In the service's Settings, set Root Directory to backend.
2

Add the MySQL database

In the same project: New → Database → MySQL. Railway creates and runs the database for you — you never install MySQL yourself. This is where all your data lives in production.
3

Set the backend environment variables

Click the backend service → Variables tab → add everything from the table in Section 8. The ${{MySQL.*}} values auto-link to your database service.
4

Set the start command

Backend service → Settings → Deploy → Custom Start Command: 
php artisan migrate --force && php artisan serve --host 0.0.0.0 --port $PORT
migrate builds all your tables in the Railway database on every deploy; serve starts the app.
5

Deploy and copy the URL

Let it build. Railway gives you a public URL like https://attendpro-backend.up.railway.app. Copy it and paste it into the APP_URL variable (replacing the placeholder).
6

Create your first admin account

The database starts empty. Open the backend service's Shell in Railway and run your seeder: 
php artisan db:seed
(Or register the first admin through the app's sign-up page once the frontend is live — it creates an admin role.)

5. Part C — Frontend on Vercel

1

Import the project

Go to vercel.com → Add New → Project → import the same GitHub repo. Set Root Directory to frontend. Vercel auto-detects Vite (Build: npm run build, Output: dist).
2

Add the one required variable

In Vercel's Environment Variables, add: 
VITE_API_URL = https://YOUR-BACKEND.up.railway.app/api
That's your Railway backend URL from Part B, with /api on the end.
3

Deploy

Click Deploy. Vercel builds and gives you a URL like https://attendpro.vercel.app. Copy it — you need it for the next part.
Build-time variable VITE_API_URL is baked into the site when it builds. If you change it later, Vercel must rebuild (it does this automatically when you change the variable or push code).

6. Part D — Connect the two (fixes CORS)

Right now the backend doesn't know your frontend's address, so the browser will block API calls (a "CORS error"). Fix it:

  1. Railway → backend service → Variables.
  2. Set FRONTEND_URL to your Vercel URL, e.g. https://attendpro.vercel.app (no trailing slash).
  3. Save — Railway redeploys automatically. Done.
The two-way link Frontend must know the backend (VITE_API_URL), and backend must know the frontend (FRONTEND_URL). Get both right and they talk freely.

7. Part E — Test the live app

Open your Vercel URL and go through this in order:

  • Log in as admin.
  • Create a branch — paste its coordinates, set a radius.
  • Create an employee and assign them a shift for today.
  • Log in as that employee → allow location → check in.
  • Confirm the check-in records, shows the branch address, and flags late/on-time correctly.

9. Common problems & fixes

SymptomCause & fix
Check-in does nothing / "location denied" Browser GPS only works over HTTPS. Make sure you're on the https:// Vercel URL, not http://. (Vercel gives HTTPS automatically.)
All API calls fail; console shows a CORS error FRONTEND_URL (Railway) or VITE_API_URL (Vercel) is wrong or has a trailing slash. Fix and redeploy.
Site loads but every request 500s APP_KEY is missing or APP_DEBUG is causing issues. Set a real APP_KEY and APP_DEBUG=false.
Can't log in — "invalid credentials", no users Database is empty — seeding didn't run. Run php artisan db:seed in the Railway shell, or sign up the first admin.
"table not found" errors Migrations didn't run. Confirm the start command includes php artisan migrate --force, or run it in the shell.
Attendance "today" is off by hours; late flags wrong APP_TIMEZONE not set to Asia/Karachi on Railway.
Large avatar / branch photo upload fails Images are stored as base64. Very large files can exceed PHP/MySQL size limits — use reasonably sized images.

10. How to make changes after deployment

This is the most reassuring part: yes, you can fix and improve things at any time. The deployment is linked to GitHub, so updating is just "push your changes."

The everyday workflow — changing code

Edit the code on your computer, then from the project root:

git add .
git commit -m "Describe what you changed"
git push

That's it. The moment you push:

  • Railway automatically rebuilds and redeploys the backend (and re-runs migrations).
  • Vercel automatically rebuilds and redeploys the frontend.

A change is usually live in 1–2 minutes. No manual re-uploading, ever.

Changing a setting (no code needed)

Wrong URL, timezone, CORS, or any environment variable? Just edit it in the Railway or Vercel dashboard. The service redeploys with the new value automatically. This covers most first-week fixes.

Your data is safe across updates

Redeploying new code does not erase the MySQL database. Your users, branches, and attendance logs stay put. The database only changes when you add a new migration (a schema change), which runs automatically via the start command.

Adding a database table or column later

  1. Create a migration locally: php artisan make:migration add_something.
  2. Write the change, test it locally with php artisan migrate.
  3. git push — Railway runs migrate --force on deploy and applies it to the production database. Existing data is preserved.

If a change breaks something — roll back

Both platforms keep your previous deployments. In the Railway or Vercel dashboard, open Deployments, find the last working one, and click Redeploy / Rollback. You're back to a good state in seconds while you investigate.

Testing before you push (recommended)

Run the app locally first so you catch mistakes before they go live:

# Terminal 1 — backend
cd backend
php artisan serve

# Terminal 2 — frontend
cd frontend
npm run dev

When it works locally, push — and it'll work in production too.

Mental model Deployment isn't "one shot." Push → check → fix → push again, as many times as you like. Treat production like a living copy of your code that updates every time you push to GitHub.
